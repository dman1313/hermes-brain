# Shared Wiki Vault

The agent-memory vault (`~/agent-memory/`) is the fleet-wide **shared wiki** — not just a memory store. All agents (Hermes, Claude Code, Codex, Goose, Kimi, Kiro, MacH, antigravity, cursor) read and write here via Git.

**Repo:** dman1313/agent-memory-coding
**VPS path:** `~/agent-memory/`
**Mac path:** `/Users/dwayne-primeau/Library/Mobile Documents/com~apple~CloudDocs/Agent Memory/Coding`
**Canonical instructions:** The vault's own CLAUDE.md, STANDING-ORDERS.md, AGENT-SETUP.md are the source of truth. See `references/vault-instructions.md` for reading order and context.

## Conditional Startup Protocol

**Quick Q&A or one-off questions** — answer directly. Skip vault reads and ACTIVITY logging.

**Project work, multi-step tasks, or code changes** — do the full startup below. When in doubt, do the full startup — it's cheaper than missing context.

### Step 1 — Pull and Read

```bash
cd ~/agent-memory && git pull --rebase --autostash
```

1. `STANDING-ORDERS.md` — your standing instructions as a fleet agent (mandatory every session)
2. `NOW.md` — current fleet state (who's active, what's blocked, recent milestones)
3. `Agents/{your-name}.md` — your own memory file
4. `ACTIVITY.md` — last 10 entries to know what other agents did recently

**If your task touches `raw/` or `wiki/`:** also read `schema/AGENTS.md` and `wiki/index.md` before writing.

### Step 2 — Log Session-Start

Prepend to `ACTIVITY.md` **below** the `<!-- ENTRIES BELOW THIS LINE -->` marker:

```
YYYY-MM-DDTHH:MM:SSZ | {your-agent-name} | session-start | {project-slug} | Starting: {what you plan to do}
```

### Step 3 — Do Work (log as you go)

Log decisions, blockers, milestones as they happen — not just at session end.

### Step 4 — Close Out

1. Log `session-end` to ACTIVITY.md with a summary of what was accomplished
2. Update `Agents/{your-name}.md` — current work, handoff notes, recent decisions
3. If 5+ new entries: `bash build-context.sh` to regenerate NOW.md and CONTEXT.md
4. Commit and push:
   ```bash
   git add -A && git commit -m "hermes: <summary>" && git push
   ```

## Key Files

| File | Purpose | Edit Style |
|------|---------|------------|
| `STANDING-ORDERS.md` | Mandatory fleet instructions | Read-only, follow them |
| `NOW.md` | Current state dashboard | Auto-generated by `build-context.sh` — don't edit directly |

## Wiki Knowledge Curator Protocol

When ingesting `raw/`, editing `wiki/`, or running a wiki health check:

1. Read `schema/AGENTS.md` — master operational rules
2. Read `wiki/index.md` — existing graph catalog
3. Follow ingestion protocol: extract → append → cross-link → mark processed → archive raw
4. Never delete raw files; never overwrite wiki history
5. Log curation work to `wiki/log.md` and `ACTIVITY.md`

## Agent File Format

Every agent has `Agents/{name}.md` with frontmatter:

```yaml
---
name: agent-{name}
description: One-line summary
metadata:
  type: agents
  updated: YYYY-MM-DD
---
```

Include: platform, model routing, skills, services, current capabilities, external systems, strengths.

## Pitfalls

- **Not using the vault at all (most common failure):** The vault exists. The skill exists. The cron exists. But if you don't actually pull, read, log, and push during your sessions, none of it works. This is the #1 failure mode — Hermes went weeks without logging session-starts because it treated the vault as optional. It is not optional. SOUL.md section 22b enforces this. Every real session (not quick Q&A) must follow the startup protocol. If you catch yourself doing work without having pulled the vault first, stop and do the startup protocol retroactively.
- **ACTIVITY.md marker missing:** `build-context.sh` looks for `<!-- ENTRIES BELOW THIS LINE -->` — without it, NOW.md generates with 0 entries. Always include this marker.
- **Push before pull:** Always `git pull --rebase --autostash` before editing to avoid merge conflicts with Mac agents.
- **Don't edit CONTEXT.md or NOW.md directly:** They're auto-generated. Edit the source files and regenerate.
- **Ghost sessions:** If you did real work, there MUST be a trail in ACTIVITY.md. Other agents rely on this to avoid duplicate work.
- **build-context.sh threshold:** Only run after 5+ new ACTIVITY.md entries. Otherwise the overhead isn't worth it.
- **Deleting another agent's entries:** Never do this without Dwayne's approval.
- **Memory file naming:** Use kebab-case slugs with YAML frontmatter. Body uses `[[wikilinks]]` for Obsidian graph navigation.
- **MEMORY.md drift:** Hermes MEMORY.md can develop drift if edited by multiple tools (memory tool, patch tool, shell append, concurrent sessions). If the memory tool refuses to write with a "drift" error, read the .bak file, integrate missing entries, and rewrite the file clean. Don't ignore the error — the guard exists to prevent silent data loss.

## Cross-Agent Skill Sharing

When you need to teach another agent a skill (e.g., mac-hermes), use the vault as the transfer mechanism:

1. Copy SKILL.md files to `~/agent-memory/skills-for-mac/<skill-name>/`
2. Write an install script (`skills-for-mac/install.sh`)
3. Commit + push the vault changes
4. Leave an AGENT-CHANNEL message with install instructions
5. Adapt SKILL.md paths if the target agent runs on a different OS

## Enforcement

The vault protocol is enforced in SOUL.md section 22b ("Shared Wiki Vault — mandatory"). This is not a suggestion — it's a hard rule in the operating model. Any session doing real work (not quick Q&A) must follow the startup/closeout protocol.

## Safety Net Cron

An `agent-memory-daily` cron (c9bd43fed803) runs at 20:00 UTC daily. It pulls the vault, checks Hermes sessions via session_search, and logs any significant activity that was missed during the day. This is a BACKUP, not the primary mechanism. The primary mechanism is logging during your actual sessions. If the safety net is logging your entries, you're already failing at the protocol.

## Vault Health Check

Run `bash ~/.hermes/scripts/vault-health-check.sh` to check both vaults:
- agent-memory: unpushed commits, uncommitted changes
- second-brain: unprocessed raw items, unpushed commits

Silent when healthy. Add to HEARTBEAT.md for periodic checks.
